# Performance Benchmarking Scripts

This directory contains scripts for performance benchmarking and regression testing of the Hello-World C program.

## Scripts

### benchmark.sh
Runs comprehensive performance benchmarks measuring:
- **Compilation times** across GCC and Clang with all optimization levels
- **Execution times** with high-precision measurements
- **Binary sizes** for each configuration

**Usage:**
```bash
./perf/benchmark.sh
```

**Output:** Creates `benchmark_results.json` with detailed performance metrics.

### set_baseline.sh
Sets the current benchmark results as the performance baseline for future comparisons.

**Usage:**
```bash
./perf/set_baseline.sh
```

**When to use:**
- After validating that current performance is acceptable
- After deliberate performance improvements
- After compiler upgrades (document in commit message)

**Output:** Creates/updates `baseline.json` with current performance metrics.

### compare_performance.sh
Compares current performance against the baseline and fails if regressions exceed thresholds.

**Usage:**
```bash
./perf/compare_performance.sh
```

**Exit codes:**
- `0`: All metrics within acceptable thresholds
- `1`: Performance regression detected

**Regression thresholds:**
- Compilation time: 20% increase
- Execution time: 50% increase
- Binary size: 10% increase

## Files

### baseline.json
Contains the baseline performance metrics that current benchmarks are compared against. This file is committed to the repository and updated when:
- Initial baseline is established
- Deliberate performance improvements are made
- Compiler versions are upgraded
- **Never** to hide performance regressions

### benchmark_results.json (gitignored)
Contains the most recent benchmark results. This file is generated by `benchmark.sh` and is not committed to version control.

## Example Workflow

```bash
# 1. Run benchmarks to measure current performance
./perf/benchmark.sh

# 2. Review results
cat perf/benchmark_results.json

# 3. If performance is acceptable, set as baseline
./perf/set_baseline.sh

# 4. Make code changes
# ... edit hello.c ...

# 5. Run benchmarks again
./perf/benchmark.sh

# 6. Check for regressions
./perf/compare_performance.sh
# This will exit with code 1 if regressions are detected
```

## CI Integration

The CI pipeline automatically runs these scripts:
1. `benchmark.sh` - Measures current performance
2. `compare_performance.sh` - Validates against baseline
3. Uploads benchmark results as artifacts (retained for 30 days)

The build fails if performance regressions exceed the configured thresholds.

## Understanding Results

### Compilation Time
Fast compilation is crucial for developer productivity:
- **GCC**: Typically 30-40ms for hello.c
- **Clang**: Typically 60-70ms (faster than the documented ~4 seconds because of caching and optimizations)
- **Factors**: Optimization level, system load, compiler version

### Execution Time
Typical execution time: ~800-850 microseconds

This includes:
- Process startup overhead
- Dynamic linking
- printf system call
- Program cleanup

### Binary Size
Varies by optimization level:
- **-O0**: ~16KB (no optimization)
- **-O1/-O2/-O3**: ~16KB (similar sizes for this simple program)
- **-Os**: ~16KB (size-optimized)

For a program this simple, optimization doesn't significantly affect binary size.

## Troubleshooting

### "bc: command not found"
Install the bc calculator:
```bash
sudo apt-get install bc
```

### Inconsistent results
Performance measurements can vary due to:
- System load
- CPU frequency scaling
- Background processes

**Solutions:**
- Close other applications
- Run benchmarks multiple times
- Use CI environment for consistent measurements

### False positive regressions
If you see false positives:
1. Run benchmarks multiple times to confirm
2. Check if thresholds need adjustment (edit `compare_performance.sh`)
3. Review what changed in the codebase or environment
